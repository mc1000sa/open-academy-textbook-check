# 새 프로젝트 시작용 복붙 프롬프트

아래 프롬프트는 새 웹앱 프로젝트를 시작하기 전에 AI에게 붙여 넣어 사용할 수 있다. 사용자는 서비스 방향만 편하게 적고, 기술 스택이나 세부 구조는 AI가 현재 프로젝트 상태를 보고 제안하게 한다.

```text
이 프로젝트를 시작하기 전에, 먼저 프로젝트 폴더 안에 AI 협업용 하네스 구조를 구성해줘.

내가 만들 프로젝트의 개발 방향은 아래와 같아.

아래 항목 중 아는 것만 채우고, 모르는 건 비워둘게. 비어 있는 항목은 네가 프로젝트 목적과 현재 폴더 상태를 보고 합리적으로 제안해줘.

- 서비스명:
- 핵심 사용자:
- 핵심 목표:
- 반드시 지켜야 할 UX 원칙:
  - 프로젝트를 만들면서 규칙을 계속 추가해줘.
- 절대 하지 말아야 할 것:
  - 내가 직접 정하기 어려우니, 현재 프로젝트 기준으로 네가 먼저 정해줘.

추가 설명:

[여기에 내가 만들고 싶은 프로젝트를 편하게 설명할게. 전문 용어를 몰라도 알아서 정리해줘.]

요구사항:

1. 현재 프로젝트 구조를 먼저 분석해줘.
2. 이미 존재하는 기술 스택, 실행 방식, 배포 방식을 네가 확인하고 쉬운 말로 설명해줘.
3. 내 개발 방향과 현재 구조가 충돌하는 부분이 있으면 알려줘.
4. 내가 기술 용어를 잘 모를 수 있으니, 필요한 기술 선택과 폴더 구조는 네가 추천해줘.
5. 아래 구조를 프로젝트에 맞게 생성하거나 갱신해줘.

프로젝트 루트/
├─ AGENTS.md
├─ RULES.md
├─ memory/
│  └─ lessons.md
├─ docs/
│  ├─ product-direction.md
│  ├─ architecture.md
│  └─ decisions/
│     └─ README.md

각 파일의 역할:

- AGENTS.md: Codex, Claude Code, Antigravity 같은 AI 에이전트가 매번 먼저 읽는 공동 작업 지침
- RULES.md: 이 프로젝트에서 반드시 지켜야 하는 개발 규칙, UI/UX 원칙, 데이터 규칙
- memory/lessons.md: 반복 실수 방지용 장기 기억 노트
- docs/product-direction.md: 제품 목표, 사용자, 핵심 기능, 성공 기준
- docs/architecture.md: 현재 폴더 구조, 기술 스택, 데이터 구조, 주요 화면, 개선 순서
- docs/decisions/README.md: 중요한 기술/제품 결정 기록 방식

작성 기준:

- 문서는 한국어로 작성해줘.
- 추상적인 말보다 이 프로젝트에 바로 적용 가능한 규칙으로 써줘.
- 현재 코드와 맞지 않는 기술 스택을 임의로 가정하지 마.
- 내가 기술 스택을 직접 지정하지 않아도, 현재 프로젝트와 목표에 맞는 선택지를 네가 제안해줘.
- 내가 명시하지 않은 프레임워크 전환은 제안만 하고 바로 적용하지 마.
- "절대 하지 말아야 할 것"은 현재 프로젝트의 위험 요소를 보고 네가 먼저 작성해줘.
- 사용자가 반복해서 설명하지 않아도 되도록, 앞으로의 AI 작업 기준이 될 만큼 구체적으로 써줘.
- 파일을 만든 뒤 어떤 파일을 만들었고, 앞으로 개발할 때 어떤 순서로 읽으면 되는지 요약해줘.
- 처음에는 빠른 목업으로 시작해도 되지만, 핵심 화면과 반복 UI가 어느 정도 보이면 AI가 먼저 Vite/React 분리 시점을 판단해서 제안해줘.
- 분리 시점이라고 판단되면 기능 추가를 잠시 멈추고, Vite/React 구조 분리 작업을 오류 없이 완료한 뒤 작업을 이어가줘.
- 분리 작업은 `src/components`, `src/lib`, `src/services`, `src/styles`, 필요 시 `src/legacy` 구조를 기준으로 해줘.
- 분리 후에는 `npm.cmd test`, `npm.cmd run build`, 로컬 개발 서버 브라우저 확인까지 검증해줘.
```

## 짧은 버전

```text
이 프로젝트의 개발 방향을 기준으로 AI 협업용 하네스 구조를 먼저 만들어줘. 나는 기술 스택이나 개발 용어를 잘 모를 수 있으니, 현재 프로젝트 구조와 목표를 보고 필요한 기술 선택, 폴더 구조, 절대 하지 말아야 할 것을 네가 먼저 제안해줘. 현재 프로젝트 구조를 분석한 뒤, `AGENTS.md`, `RULES.md`, `memory/lessons.md`, `docs/product-direction.md`, `docs/architecture.md`, `docs/decisions/README.md`를 프로젝트에 맞게 작성해줘. 문서는 한국어로, 추상론이 아니라 이 프로젝트에서 바로 지킬 수 있는 규칙으로 써줘. 현재 코드와 맞지 않는 기술 스택은 임의로 가정하지 말고, 충돌 사항은 먼저 알려줘. 처음 목업으로 시작하더라도 핵심 화면과 반복 UI가 보이면 AI가 먼저 Vite/React 분리 시점을 제안하고, 승인 후 `src/components`, `src/lib`, `src/services`, `src/styles`, 필요 시 `src/legacy` 구조로 분리한 뒤 테스트/빌드/브라우저 검증까지 완료해줘.
```

## Vite/React 분리 시점 판단 프롬프트

목업을 어느 정도 만든 뒤 아래 프롬프트를 추가로 붙여 넣으면 된다.

```text
지금 목업이 Vite/React 분리 작업을 진행할 시점인지 판단해줘.

다음 기준으로 봐줘.

- 단일 HTML/JS/CSS 파일이 커지고 있는가?
- 화면이 3개 이상 생겼는가?
- 반복되는 버튼, 카드, 표, 입력 폼, 모달 패턴이 보이는가?
- Firebase/API/계산 로직/화면 렌더링이 한 파일에 섞이고 있는가?
- 앞으로 보고서, 차트, 권한, 누적 계산처럼 복잡도가 커질 기능이 예정되어 있는가?
- 실제 데이터를 저장하거나 불러오기 시작했는가?

분리할 시점이면 이유를 설명하고, 내 승인 후 Vite/React 구조로 분리해줘.
분리 구성은 `src/components`, `src/lib`, `src/services`, `src/styles`, 필요 시 `src/legacy`를 기본으로 해줘.
기존 화면은 깨지지 않게 보존하고, `npm.cmd test`, `npm.cmd run build`, 로컬 개발 서버 브라우저 확인까지 완료한 다음 다음 기능 개발로 넘어가줘.
```

# 열린학원 수학교재점검 시스템 개발 규칙

이 파일은 이 프로젝트를 개발할 때 AI와 개발자가 반드시 지켜야 하는 기준이다. 작업 전 항상 먼저 읽고, 충돌하는 요구가 있으면 사용자에게 확인한다.

## 1. 제품 핵심 원칙

- 강사가 무조건 편하게 입력해야 한다.
- 교재 점검은 버튼식, 선택식, 자동 계산식 입력을 우선한다.
- 타이핑은 메모, 예외 사유, 검색처럼 꼭 필요한 경우에만 허용한다.
- 학부모 보고서는 아이의 교재 관리 상태를 한눈에 보여주어야 한다.
- 반별 보고서는 반 전체 완성도와 6요소 평균을 숫자로 비교할 수 있어야 한다.

## 2. 핵심 평가 요소

교재 점검 기록은 아래 6요소를 누적 관리할 수 있어야 한다.

1. 과제 수행률
2. 수학식 논리및잘씀정도
3. 채점약속
4. 수업태도
5. 숙지력
6. 심화응용력

과제 수행률은 교재 범위와 미완료 페이지를 기준으로 자동 계산한다. 나머지 5요소는 강사가 버튼식 점수로 입력한다.

## 3. 보고서 규칙

- 학부모 보고서는 단원별 완성도를 가로 막대그래프로 보여준다.
- 학생별 6요소는 반 평균과 학생 누적 수치를 오버랩해서 보여준다.
- 6요소 비교 시 레이더 차트 또는 동일한 의미의 6각형 시각화를 우선한다.
- 반별 전체 보고서에는 학생별 6요소와 전체 평균이 표로 표시되어야 한다.
- 인쇄/PDF 저장 화면은 `@media print`를 기준으로 깨짐 없이 보여야 한다.

## 4. 현재 기술 스택

- 프론트엔드 실행 구조: Vite + React
- 호스팅: Firebase Hosting
- 데이터베이스: Firebase Firestore
- 인증: Firebase Authentication 익명 로그인 기반
- 현재 전환 상태: 기존 앱 코드가 `src/legacy/legacyApp.js`와 `src/styles/legacy.css`로 분리되어 React 컴포넌트에서 직접 마운트된다.
- 스타일: 신규 코드는 `src/styles`와 컴포넌트 CSS를 사용하고, 레거시 화면은 기존 Tailwind CDN과 인라인 CSS를 유지한다.

주의: 기존 기능은 아직 `src/legacy/legacyApp.js`에 많이 남아 있다. 새 기능은 가능한 `src/components`, `src/lib`, `src/styles`에 추가하고, 레거시 코드는 점진적으로 이동한다.

## 5. 데이터 모델 원칙

- 기존 Firestore 컬렉션 이름과 필드 의미를 함부로 바꾸지 않는다.
- 새 필드는 기존 기록을 읽을 수 있도록 기본값과 하위 호환 처리를 둔다.
- 점검 기록은 학생, 반, 교재, 날짜, 범위, 완료율, 6요소 평가가 함께 추적되어야 한다.
- 보고서 계산은 저장된 원천 기록에서 재계산 가능해야 한다.

## 6. UI/UX 원칙

- 강사용 화면은 수업 직후 빠른 입력이 목적이다.
- 자주 쓰는 평가는 큰 버튼, 칩, 세그먼트 컨트롤로 제공한다.
- 상태 요약은 숫자와 막대그래프를 함께 제공한다.
- 색상은 열린학원 브랜드에 맞는 고급스러운 블루/인디고 계열을 중심으로 하되, 완료/주의/미완료 상태는 명확히 구분한다.
- 모바일에서도 표와 보고서가 읽혀야 한다.

## 7. 개발 워크플로우

- 변경 전 현재 구조와 관련 함수 위치를 먼저 확인한다.
- 새 프로젝트에서 먼저 단일 HTML, 정적 목업, 또는 빠른 프로토타입으로 뼈대를 만든 경우, 기능 추가를 계속하기 전에 Vite/React 분리 가능 시점을 판단한다.
- 목업의 핵심 화면, 주요 상태 흐름, 반복 UI 패턴이 보이기 시작하면 AI가 먼저 "이제 Vite/React 분리 작업을 진행할 시점"이라고 제안한다.
- 사용자가 승인하면 기능 추가를 잠시 멈추고, 오류 없이 Vite/React 분리 작업을 완료한 뒤 다음 기능 개발을 이어간다.
- 작은 단위로 수정하고, 기존 기능을 깨지 않았는지 확인한다.
- 배포 전 가능한 검증:
  - 정적 파일 문법 확인
  - 브라우저에서 주요 화면 확인
  - Firebase Hosting 설정 확인
- 새로운 시행착오와 사용자 선호는 `memory/lessons.md`에 기록한다.
- 작업의 의미 있는 묶음이 완료되었을 때, 사용자가 그대로 복사해 사용할 수 있는 형식으로 큰 커밋 제목과 본문 내용을 정리하여 리포트하고, 배포 명령은 독립된 단일 코드블록(예: `firebase deploy --only hosting`)으로 가독성 높게 표시한다.

## 8. Vite/React 분리 제안 기준

AI는 다음 조건 중 2개 이상이 보이면 분리 작업을 제안한다.

- `index.html` 또는 단일 파일 안에 HTML, CSS, JavaScript가 함께 커지고 있다.
- 화면이 3개 이상이거나, 로그인/대시보드/입력/보고서처럼 역할이 다른 화면이 생겼다.
- 같은 버튼, 카드, 표, 입력 폼, 모달 패턴이 반복된다.
- Firebase, API, 계산 로직, 화면 렌더링 로직이 한 파일에 섞이기 시작했다.
- 앞으로 보고서, 차트, 권한, 누적 계산처럼 복잡도가 커질 기능이 예정되어 있다.
- 단순 목업을 넘어 실제 데이터를 저장하거나 불러오기 시작했다.

분리 작업을 제안할 때는 다음을 함께 설명한다.

- 지금 분리해야 하는 이유
- 분리하지 않고 계속 개발할 때 생길 위험
- 분리 후 폴더 구조
- 기존 화면을 보존하는 전환 방식
- 검증 명령과 브라우저 확인 방법

## 9. 표준 Vite/React 분리 구성

목업에서 Vite/React로 전환할 때 기본 구조는 아래를 따른다.

```text
프로젝트 루트/
├─ index.html
├─ package.json
├─ vite.config.js
├─ firebase.json
├─ public/
│  └─ 정적 이미지와 404.html
└─ src/
   ├─ main.jsx
   ├─ App.jsx
   ├─ components/
   │  └─ 화면과 UI 컴포넌트
   ├─ lib/
   │  └─ 계산 함수, 데이터 변환, 공통 유틸
   ├─ services/
   │  └─ Firebase/API 접근 코드
   ├─ styles/
   │  └─ 전역 CSS와 화면별 CSS
   └─ legacy/
      └─ 아직 React 컴포넌트로 옮기지 못한 기존 코드
```

분리 순서:

1. Vite/React 실행 구조를 만든다.
2. 기존 목업 화면이 깨지지 않도록 먼저 보존한다.
3. CSS를 `src/styles`로 분리한다.
4. JavaScript를 `src/legacy` 또는 `src/lib`로 분리한다.
5. Firebase/API 코드를 `src/services`로 분리한다.
6. 계산 로직을 `src/lib`의 테스트 가능한 순수 함수로 옮긴다.
7. 반복 UI부터 `src/components`로 옮긴다.
8. `npm.cmd test`, `npm.cmd run build`, 브라우저 확인을 통과한 뒤 기능 개발을 재개한다.

## 10. 향후 구조 전환 기준

`src/legacy/legacyApp.js`의 기존 코드가 계속 커지지 않도록 다음 순서로 전환한다.

1. CSS, Firebase 접근, 계산 로직을 별도 파일로 분리
2. 테스트 가능한 순수 계산 함수 분리
3. 보고서 화면 React 컴포넌트화
4. 점검 입력 화면 React 컴포넌트화

전환 목적은 기술 과시가 아니라 유지보수성, 보고서 안정성, 입력 속도 개선이어야 한다.

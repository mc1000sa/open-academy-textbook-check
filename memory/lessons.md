# 열린학원 수학교재점검 시스템 누적 학습 노트

이 파일은 개발 과정의 시행착오, 사용자 선호, 반복 실수를 기록하는 장기 기억 저장소다. 새 세션을 시작하면 `RULES.md`와 함께 반드시 읽는다.

## 1. 사용자 선호와 제품 방향

- 사용자는 프로젝트 시작 전에 개발 방향을 먼저 정하고, 그 방향을 프로젝트 폴더 안의 문서와 폴더 구조로 고정하고 싶어 한다.
- 강사용 교재 점검 입력은 빠르고 직관적이어야 한다. 타이핑보다 버튼 클릭, 선택식 점수, 자동 계산을 우선한다.
- 학부모 보고서는 말로 길게 설명하기보다 막대그래프와 6각형 비교처럼 한눈에 이해되는 시각화를 선호한다.
- 반별 전체 보고서는 감이 아니라 수치로 반의 교재 완성도와 학생별 상태를 비교해야 한다.
- 사용자는 앞으로 새 웹앱 프로젝트를 시작할 때 `AGENTS.md`, `RULES.md`, `memory/lessons.md`, `docs/` 구조를 먼저 만들고 진행하고 싶어 한다.
- 사용자는 처음에는 빠르게 목업 웹을 만들 수 있지만, 핵심 화면과 반복 UI가 보이면 AI가 먼저 Vite/React 분리 시점을 제안해주길 원한다.
- 사용자는 분리 시점을 직접 판단하기 어려워하므로, AI가 "지금 분리해야 하는 이유"와 "분리 후 구조"를 설명하고 승인 후 오류 없이 분리한 다음 작업을 이어가야 한다.
- 관리자 페이지는 항상 섹션별로 카드 형식으로 관리자 메뉴를 정리하고, 카드는 최초 진입 시 접기(collapsed) 형식으로 구성하여 불필요한 스크롤을 줄인다.
- 시스템 내 모든 주요 설정 및 텍스트(예: 로그인/스플래시 화면 등)는 하드코딩하지 않고 관리자 메뉴에서 실시간 조정 및 DB 연동이 가능하도록 반영한다.

## 2. 현재 프로젝트에서 확인한 사실

- 현재 저장소는 Vite/React 구조로 전환되었다.
- 기존 앱 코드는 `src/legacy/legacyApp.js`와 `src/styles/legacy.css`로 분리되었고, React 컴포넌트가 직접 마운트한다.
- 새 작업 위치는 `src/components`, `src/lib`, `src/styles`다.
- `firebase.json`의 hosting public 경로는 Vite 빌드 결과물인 `dist`다.
- 현재 루트 `index.html`은 Tailwind CDN과 Font Awesome CDN을 로드하고, Firebase는 npm 패키지를 사용한다.
- 기존 샘플 `RULES.md`에는 React/Vite와 Tailwind 금지 규칙이 있었지만, 이 프로젝트에는 그대로 적용하면 안 된다.

## 3. 자주 조심할 점

- 기존 Firestore 데이터와 컬렉션을 깨지 않도록 새 필드는 하위 호환으로 추가한다.
- 보고서 출력 기능은 화면 표시와 인쇄/PDF 저장을 모두 고려한다.
- 사용자가 "편하게 입력"을 강조하면, 입력 필드 추가보다 버튼식 UX를 먼저 설계한다.
- 단원별 완료율, 학생 누적 점수, 반 평균은 계산 기준을 문서화하고 재현 가능하게 만든다.
- 단일 HTML/목업 파일이 커지는 동안 기능을 계속 추가하면 나중에 분리 오류가 커지므로, 로그인/대시보드/입력/보고서 같은 주요 화면이 보이는 즉시 분리 제안을 검토한다.
- **브라우저 기본 팝업(alert, confirm, prompt)은 절대 사용하지 않는다.** 반드시 프로젝트 톤앤매너에 맞는 커스텀 모달(`showModalAlert`, `showModalConfirm`, `showModalPrompt`)을 사용한다. 이 규칙을 어기면 즉시 교체한다.
- 관리자 메뉴 내 카드(아코디언) UI는 프로젝트 다크 테마(`bg-slate-900`, `border-slate-800`, `text-slate-200` 등)와 완전히 일치해야 한다. 흰색 배경(`bg-white`)과 밝은 테마를 카드 내부에 하드코딩하지 않는다.
- setupView.js, bookSetupView.js 등 뷰 파일의 버튼에는 반드시 명확한 hover 색상 변화(`hover:bg-xxx hover:text-white transition-colors`)를 추가한다.
- 반(class) "수정" 버튼은 폼에 기존 값을 채우고 1번 아코디언을 열어주는 방식으로 구현한다. 별도 수정 모달을 만들지 않는다.
- 과제 점검 개선에서는 과거 점검 원본을 수정하지 않고, 새 점검 기록의 `carryoverResolutions`에 지난 미완료 회수 이력을 저장한다.
- 6요소 명칭은 `과제 수행률`, `풀이 표현력`, `채점 성실도`, `수업 태도`, `개념 이해도`, `응용 해결력`을 사용한다.
- 이번 회차 과제 수행률과 지난 미완료 회수율은 합산하지 않고 별도 지표로 표시한다.

## 4. 업데이트 규칙

- 사용자가 새 규칙을 알려주면 이 파일에 즉시 반영한다.
- 해결한 버그, 깨진 UI 원인, Firebase 데이터 이슈는 짧게 기록한다.
- 오래되어 틀린 내용은 남겨두지 말고 수정한다.
- 사용자는 크레딧 소모를 매우 신경 쓴다. 큰 기능 작업 후에는 무거운 전체 빌드/전체 브라우저 검증보다 `node --check` 같은 가벼운 확인을 우선하고, 시간이 오래 걸리거나 크레딧이 큰 검증은 사용자에게 확인 요청을 먼저 한다.
- 전체 디자인 고급화는 화려하고 무거운 애니메이션보다 Apple 스타일처럼 여백, 타이포그래피, 미세한 hover/active 반응, 부드러운 Aurora 배경을 활용한 깔끔한 프로페셔널 톤을 선호한다.
- 교재 단원 통계는 교재 목차명과 표준 소단원 ID를 분리해서 관리한다. 표준단원은 너무 세부 개념으로 쪼개지 말고 시중교재 목차 수준에 가깝게 두며, 교재 단원 하나가 여러 표준단원 ID를 연결할 수 있어야 한다.
- 교재 `보관`은 교재 자산 자체를 숨겨두는 개념이고, 반별 `진행 종료`는 특정 반에서만 해당 교재 점검을 끝내는 개념이다. 같은 교재는 다른 반에서 계속 진행할 수 있어야 하므로 반-교재 연결(`classBooks`)의 상태로 관리한다.

## 5. 프로젝트 현재 구현 및 분석 상태 (2026-05-23 기준)

- **아키텍처 및 구조**: Vite/React 구조로 변환 완료. 레거시 메인 코드는 `src/legacy/legacyApp.js`(3,077줄)와 `src/styles/legacy.css`에 분리 보관 중이며, React 컴포넌트가 직접 이를 마운트해 구동 중임.
- **핵심 뷰 구현 현황 (src/legacy/views/)**:
  - `loginView.js` (로그인 및 스플래시 화면)
  - `layoutView.js` (공통 레이아웃 구조)
  - `dashboardView.js` (대시보드 메인)
  - `inspectionsView.js` (점검 입력 화면)
  - 기능 구현 자체는 대부분 완료되었으나, 순수 React 컴포넌트(`src/components`, `src/lib`, `src/styles`)로의 완전한 이관은 아직 진행 중/미완성 상태임.
- **문서 불일치 항목**: `docs/architecture.md` 문서와 현재 파일 구조 간에 불일치가 있음. `adminStudentMaintenance.js`, `remarkTemplates.js`, `standardUnits.js` 등이 실제 코드로 구현되었으나 아키텍처 문서에는 미반영됨.
- **테스트 누락**: 대부분의 주요 뷰에 `.test.js` 테스트 파일이 마련되어 있으나, `studentPortalView` 및 `remarkTemplates` 관련 테스트가 현재 누락되어 있음.
